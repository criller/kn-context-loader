# 业务知识网络通俗解释

## 什么是业务知识网络？

### 核心定义
**业务知识网络 = 概念定义 + 资源**

- **概念定义**（本体建模）：基于本体论，定义业务中的概念是什么
  - 概念定义包含：定义对象类、定义关系类、定义行动类等
- **资源**：实际的数据、逻辑、工具

### 快速理解：医疗知识网络示例

以医疗知识网络为例，业务知识网络是这样工作的：

1. **概念定义（本体建模）**：
   - **定义对象类**：定义"疾病"对象类，说明"疾病"这类实体有什么属性（疾病ID、疾病名称、ICD编码等）
   - **定义对象类**：定义"症状"对象类，说明"症状"这类实体有什么属性（症状ID、症状名等）
   - **定义对象类**：定义"药物产品"对象类，说明"药物产品"这类实体有什么属性（药品名、规格、剂型等）
   - **定义关系类**：定义"疾病 → 有症状 → 症状"关系类，说明"疾病"和"症状"之间可以有"有症状"这种关系
   - **定义关系类**：定义"疾病 → 可用药物 → 药物"关系类，说明"疾病"和"药物"之间可以有"可用药物"这种关系

2. **映射资源**：
   - "疾病"对象类映射到数据库中的疾病表（通过数据视图）
   - "疾病-症状"关系类映射到疾病症状关系表（通过数据视图）

3. **具体事实（实例）**：
   - 数据库中的"感冒"、"高血压"、"糖尿病"等是"疾病"对象类的具体实例
   - 数据库中的"发热"、"咳嗽"等是"症状"对象类的具体实例
   - 数据库中的"阿司匹林片剂100mg"是"药物产品"对象类的具体实例
   - 数据库中的"感冒 → 有症状 → 发热"是"疾病 → 有症状 → 症状"关系类的具体实例

4. **AI使用**：
   - 当用户问"感冒有哪些症状？"时，AI知道：
     - "感冒"是"疾病"对象类的一个具体实例
     - 可以通过"疾病 → 有症状 → 症状"关系类查询
     - 查询映射到的"疾病症状关系表"数据视图
     - 返回具体的关系实例：感冒 → 有症状 → 发热、咳嗽、流鼻涕等

## 重要概念区分：概念定义 vs 具体实例

### 对象类 vs 对象实例

| 概念 | 定义 | 医疗知识网络示例 |
|------|------|----------------|
| **对象类** | 概念定义的一部分，定义某类实体有什么属性 | "疾病"对象类、"症状"对象类、"药物产品"对象类 |
| **对象实例** | 具体事实，是对象类的一个具体实例 | "感冒"（疾病对象类的实例）、"发热"（症状对象类的实例）、"阿司匹林片剂100mg"（药物产品对象类的实例） |

### 关系类 vs 关系实例

| 概念 | 定义 | 医疗知识网络示例 |
|------|------|----------------|
| **关系类** | 概念定义的一部分，定义两类实体之间可以有什么关系类型 | "疾病 → 有症状 → 症状"关系类、"疾病 → 可用药物 → 药物"关系类 |
| **关系实例** | 具体事实，是关系类的一个具体实例 | "感冒 → 有症状 → 发热"（关系类的实例）、"高血压 → 可用药物 → 阿司匹林"（关系类的实例） |

### 行动类 vs 行动实例

| 概念 | 定义 | 医疗知识网络示例 |
|------|------|----------------|
| **行动类** | 概念定义的一部分，定义可以执行什么操作类型 | "开处方"行动类、"发送诊断报告"行动类（医疗知识网络中暂未定义） |
| **行动实例** | 具体操作，是行动类的一个具体执行 | "给患者张三开处方"（行动类的实例） |

**核心理解**：
- **概念定义（本体建模）**：包含定义对象类、定义关系类、定义行动类等，告诉AI"是什么类型"、"有什么属性"、"可以有什么关系"、"可以执行什么操作"
- **具体实例（具体事实）**：告诉AI"具体是什么"、"具体有哪些值"、"具体有什么关系"、"具体执行了什么操作"

## 核心组成：用医疗知识网络举例

### 1. 概念定义（本体建模）

**概念定义（本体建模）就是基于本体论，定义业务中的概念是什么。**

**概念定义包含**：
- **定义对象类**：定义某类实体有什么属性
- **定义关系类**：定义两类实体之间可以有什么关系类型
- **定义行动类**：定义可以执行什么操作类型

**重要区分**：
- **对象类**：概念定义的一部分，比如"疾病"对象类、"症状"对象类、"药物产品"对象类
- **对象实例**：具体事实，比如"感冒"（疾病对象类的一个实例）、"发热"（症状对象类的一个实例）、"阿司匹林片剂100mg"（药物产品对象类的一个实例）

以医疗知识网络为例，概念定义包含三类：

#### 定义对象类：定义业务实体的类型

**对象类是概念定义的一部分，不是具体事实。**

在医疗知识网络中，通过概念定义定义了以下对象类：

- **疾病类**（disease）
  - **这是概念定义**：定义"疾病"这类实体有什么属性
  - 属性定义：疾病ID、疾病名称、ICD编码、是否传染病、治疗费用、治愈率等
  - 映射：这些属性定义映射到数据视图中的具体字段（如`disease_name`属性映射到数据库的`disease_name`字段）
  - **具体实例**：数据库中的"感冒"、"高血压"、"糖尿病"等都是"疾病"类的具体实例

- **症状类**（symptom）
  - **这是概念定义**：定义"症状"这类实体有什么属性
  - 属性定义：症状ID、症状名
  - 映射：映射到症状数据表
  - **具体实例**：数据库中的"发热"、"咳嗽"、"流鼻涕"等都是"症状"类的具体实例

- **药物产品类**（drug_product）
  - **这是概念定义**：定义"药物产品"这类实体有什么属性
  - 属性定义：药品名、规格、剂型、批准文号、生产厂商等
  - 映射：映射到药物产品数据表
  - **具体实例**：数据库中的"阿司匹林片剂100mg"、"阿司匹林肠溶片"等都是"药物产品"类的具体实例

- **科室类**（department）
  - **这是概念定义**：定义"科室"这类实体有什么属性
  - 属性定义：科室ID、科室名
  - 映射：映射到科室数据表
  - **具体实例**：数据库中的"心内科"、"呼吸科"等都是"科室"类的具体实例

#### 定义关系类：定义实体之间的关系类型

**关系类是概念定义的一部分，不是具体事实。**

在医疗知识网络中，通过概念定义定义了以下关系类：

- **疾病 → 有症状 → 症状**（has_symptom）
  - **这是概念定义**：定义"疾病"类和"症状"类之间可以有"有症状"这种关系类型
  - 映射：通过"疾病症状关系表"数据视图来存储这种关系
  - **具体实例**：数据库中的"感冒 → 有症状 → 发热"、"感冒 → 有症状 → 咳嗽"等都是这个关系类的具体实例

- **疾病 → 可用药物 → 药物**（cures）
  - **这是概念定义**：定义"疾病"类和"药物"类之间可以有"可用药物"这种关系类型
  - 映射：通过"疾病可用药物关系表"数据视图来存储
  - **具体实例**：数据库中的"高血压 → 可用药物 → 阿司匹林"是这个关系类的具体实例

- **疾病 → 检查项 → 检查项目**（checks）
  - **这是概念定义**：定义"疾病"类和"检查项目"类之间可以有"检查项"这种关系类型
  - 映射：通过"疾病检查项目关系表"数据视图来存储
  - **具体实例**：数据库中的"糖尿病 → 检查项 → 血糖检测"是这个关系类的具体实例

- **疾病 → 分诊科室 → 科室**（designated_to）
  - **这是概念定义**：定义"疾病"类和"科室"类之间可以有"分诊科室"这种关系类型
  - 映射：通过"疾病对应分诊科室"数据视图来存储
  - **具体实例**：数据库中的"心脏病 → 分诊科室 → 心内科"是这个关系类的具体实例

- **药物 → 可选药品 → 药物产品**（is_a_drug）
  - **这是概念定义**：定义"药物"类和"药物产品"类之间可以有"可选药品"这种关系类型
  - 映射：通过"药物产品归属药物关系表"数据视图来存储
  - **具体实例**：数据库中的"阿司匹林 → 可选药品 → 阿司匹林片剂100mg"是这个关系类的具体实例

#### 定义行动类：定义可以执行的操作类型

**行动类是概念定义的一部分，不是具体操作。**

在医疗知识网络中，行动类为空（action_types: []），但在其他业务场景中可能通过概念定义包括：
- **定义"开处方"行动类**：定义"开处方"这种操作类型，需要什么输入参数，会执行什么操作
- **定义"发送诊断报告"行动类**：定义"发送诊断报告"这种操作类型
- **定义"预约检查"行动类**：定义"预约检查"这种操作类型

**具体操作实例**：当AI实际执行"给患者开处方"时，这是"开处方"行动类的一个具体操作实例

### 2. 资源

**资源是概念定义背后实际的数据、逻辑和工具。**

在医疗知识网络中：

- **数据资源**：
  - 每个对象类都映射到一个数据视图（data_view）
  - 例如："疾病"对象类映射到ID为`d3ccqbrrm8ppom9o972g`的数据视图
  - 这个数据视图对应数据库中的疾病表，包含疾病ID、疾病名称等字段

- **逻辑资源**：
  - 可以定义计算规则（如"重症疾病 = 治愈率 < 30%"）
  - 在医疗知识网络中，主要通过数据属性来体现

- **行动资源**：
  - 医疗知识网络中暂未定义行动类，但在实际应用中可能包括：
  - 开处方的工具、发送报告的API等

## 为什么需要业务知识网络？

### 问题场景
企业有多个系统：
- CRM系统（客户管理）
- ERP系统（订单管理）
- 财务系统
- 营销工具

**问题**：
- AI不知道"客户"在这些系统中分别指什么
- AI不知道"客户"和"订单"有什么关系
- AI不知道可以执行哪些操作

### 解决方案：通过概念定义统一理解
1. **定义概念**：明确"客户"、"订单"等概念的定义
2. **定义关系**：明确"客户"和"订单"之间的关系
3. **定义操作**：明确可以执行哪些操作（如"发送营销短信"）
4. **映射资源**：将概念定义映射到实际的数据、逻辑、工具

## 三层架构

### 第一层：通用业务知识网络
定义跨行业通用的基础概念：
- **对象类**：人员、客户、产品
- **关系类**：属于、关联、影响
- **行动类**：基础操作

### 第二层：行业知识网络
在通用概念基础上，定义行业专属概念：
- **金融行业**：风控指标、风险评估等概念
- **零售行业**：供应链节点、库存管理等概念

### 第三层：客户自定义知识网络
企业根据自身业务，定义个性化概念：
- 特定业务流程的概念
- 特定客户分群策略的概念

## 核心功能（三大能力）

### 1. 本体建模：概念定义
**作用**：建立概念定义体系（基于本体论）
- **定义对象类**：定义"疾病"对象类是什么、"症状"对象类是什么
- **定义关系类**：定义"疾病"对象类和"症状"对象类之间有什么关系类型
- **定义行动类**：定义"开处方"行动类是什么操作类型
- **资源映射**：将概念定义映射到实际的数据表、计算规则、工具（数据表中存储的是具体的对象实例和关系实例）

### 2. 本体索引：提升查询效率
**作用**：为概念定义建立索引，加快查询速度
- **全量构建**：一次性为所有概念类建立索引
- **增量构建**：只更新变化的概念类索引
- **分组构建**：按概念分组独立构建索引

### 3. 本体交互：提供查询和操作能力
**作用**：让AI能够查询概念定义和具体实例，执行操作
- **查询能力**：
  - 查询概念定义："疾病"类的定义是什么？（查询对象类）
  - 查询对象实例："有哪些疾病？"（查询"疾病"类的具体实例，如"感冒"、"高血压"）
  - 查询关系类定义："疾病 → 有症状 → 症状"关系类的定义是什么？（查询关系类）
  - 查询关系实例："感冒有哪些症状？"（查询"疾病 → 有症状 → 症状"关系类的具体实例）
- **操作能力**：
  - 执行行动类："给患者开处方"（执行"开处方"行动类的一个具体操作实例）
  - 触发业务操作："生成月度销售报表"

## 实际应用场景：医疗知识网络示例

### 场景：AI医疗助手查询疾病信息

**用户提问**："感冒有哪些症状？应该用什么药？"

#### 没有业务知识网络的情况：
- AI不知道"感冒"是什么概念
- AI不知道"症状"是什么概念
- AI不知道"感冒"和"症状"有什么关系
- AI不知道如何查询这些信息

#### 有了业务知识网络的情况：

**步骤1：AI理解概念定义（对象类、关系类）**
- AI查询"疾病"对象类，理解"疾病"类是什么（概念定义）
- AI查询"症状"对象类，理解"症状"类是什么（概念定义）
