# 场景：关系与路径查询（Relation / Path）

目标（聚焦用户与业务）
- 关系确认：判断两个实体之间是否存在某种关系（一般为一跳），返回相邻边与节点。
- 路径查询（M2）：在多跳范围内查找从源到目的的路径，可约束关系类型、路径长度、方向；可返回最短路径或全部匹配路径（受限）。

用户故事（化工示例，可替换为你们领域）
- RP-001 企业与园区的关系
  - “企业A是否坐落在园区B？”
  - 按TBox的关系路径：enterprise → park（enterprise_2_park，正向，一跳）
  - 期望：若存在该边，返回边与节点；否则明确说明不存在。
- RP-002 企业与专利关系
  - “企业A是否拥有专利P001？”
  - 按TBox的关系路径：enterprise → patent（enterprise_2_patent，正向，一跳）
  - 期望：返回企业到专利的一跳关系与节点。
- RP-003 路径：企业→物料→反应→产物【M2】
  - “企业A生产的物料参与了哪些反应，并最终产出哪些物料？”
  - 按TBox的关系路径：enterprise → material（enterprise_2_material） → reaction（material_2_reaction） → material（reaction_2_material，默认各跳为正向）
  - 期望：返回符合约束的一条最短路径或路径集合。
- RP-004 反向：园区包含企业（反向一跳）
  - “园区B里有哪些企业？”
  - 按TBox的关系路径：park ← enterprise（enterprise_2_park，反向，一跳）
  - 期望：返回与园区B存在 enterprise_2_park 入向关系的企业节点集与边清单。
- RP-005 反向：专利的拥有者（反向一跳）
  - “专利P001由哪些企业拥有？”
  - 按TBox的关系路径：patent ← enterprise（enterprise_2_patent，反向，一跳）
  - 期望：返回拥有该专利的企业列表与对应边。

基础能力（文本版）
- 支持一跳关系判断与返回相邻节点/边清单；
- 支持限定关系类型、方向（正向/反向/双向）、深度；默认策略A（一跳/路径默认正向）；
- （M2）支持路径探索与最短路径计算；
- 对不支持的请求能够给出替代建议（如先子图扩展，再在客户端匹配路径）。

注意事项
- 当目标实体未指定时，可返回所有匹配的一跳关系用于浏览；
- 路径能力（M2）可能受性能与规模限制，需要明确约束与边界。
